### Action 8.2 — AOT Kernel Cache (MACA Shader Cache)
**Decision: KEEP**

**Key discovery:** MACA has a built-in persistent JIT kernel cache at `~/.metax/shadercache/` (86 MB, 7 `.cache` files, version-tagged `_3.3.0.15.cache`). This was populated on the very first run (April 25). The Phase 1 baseline (10 tok/s, 135s cold start) was measured with an *empty* cache.

**Results (warm cache, April 29):**
- Image cold start (post-restart): **135.2s → 4.28s (-97%)**
- Image warm: **4.38s → 3.44s (-21%)**
- Text decode: **10 tok/s → 46 tok/s (+360%)**

**Recommendation:**
- Back up `~/.metax/shadercache/` before any MACA upgrade
- Add a post-ready warmup script that runs 2–3 inference requests immediately after server start

Doc: `task2-optimization/profiling/actions/action-8.2-aot-kernel-cache.md`

### Action 8.4 — RMSNorm + Linear Projection Fusion
**Decision: FUTURE WORK (BLOCKED at config level)**

**Investigation (2026-04-29):**

Inspected FastDeploy 2.5 source + MetaX custom device `.so` symbols to determine if a fused RMSNorm+Linear kernel exists.

Findings:
1. FastDeploy already uses `paddle.incubate.nn.functional.fused_rms_norm` (not raw RMSNorm)
2. MetaX custom device registers `fused_rms_norm_ext_metax_gpu_ALL_LAYOUT` ✅
3. MetaX custom device does **NOT** register `norm_linear`, `fused_norm_linear`, or `rms_norm_gemv` ❌
4. MetaX backend in FastDeploy has `attention/` and `moe/` overrides, but **no `linear.py` or `normalization.py`** ❌
5. The Phase 1 profiler showed `phi::RmsNormBlockSMemImpl<bf16>` = the MACA JIT translation of `fused_rms_norm_ext_metax_gpu` (MACA runs CUDA PTX via JIT; kernel names retain CUDA naming)

**Conclusion:** `fused_rms_norm` already dispatches to MetaX-accelerated kernel. A deeper RMSNorm+GEMV fusion (single kernel) doesn't exist in the current stack. Implementation would require custom MACA kernel (est. 3–5 days effort, +3–5% expected gain).

Doc: `task2-optimization/profiling/actions/action-8.4-rmsnorm-linear-fusion.md`
